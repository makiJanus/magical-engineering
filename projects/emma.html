Emma: A robot girl that likes to talk

Overview

▶ Watch the demo

Emma, as I like to call her, is a social robot designed to interact with children in a natural and engaging way. She has a friendly and approachable appearance, with a soft and colorful exterior that is designed to be visually appealing to children. Emma is equipped with a variety of sensors, including cameras, microphones, and touch sensors, which allow her to perceive her environment and interact with children in a more natural and playfulway.

So, I spent around 2 months working on this project, from cad design, to assembling and testing. I designed the robot using fusion 360, where the face, that allows more natural intreractions, is made out of silicone! I used a 3D printed mold to cast the face, and then I assembled the rest of the robot using 3D printed parts and some off-the-shelf components. The robot is controlled by a raspberry pi 3B+ and coded in python, using the GPIO to read the sensors, mange the screens, and control the servos.

The result is a social robot works, it can interact with children in a natural way and the overall robot can be used as platform for continuous development and research in this field. I am really happy with the result, and I hope to continue working on this project in the future, adding more features and improving the design.

Design

The appeance of the robot is of most importance, since it is the first thing that children will see and interact with. I wanted to design a robot that is firendly and approachable, these are some of the main references I used as concept design:

Of course one important aspect is the face itself, soft and skin like material is a must, so I decided to use tinted silicone that can be casted in a 3D printed mold, this way I can achieve a more natural and appealing look, and also allows for more natural interactions with the children, since it is soft and flexible. The rest of the robot is made out of 3D printed parts covered on transparent resin for further protenction (mostly for the painting), that are designed to be easy to assemble and disassemble, this way I can easily make changes and improvements to the design in the future.

Of course not all is 3D printed, the electronics are also an important part of the design, we already know that the robot is controlled by a raspberry pi, due the quiantity of elements connected to it, it was necessary to design a custom circuit board that link all the components together, this way I can have a more organized and efficient design, and also allows for easier troubleshooting and maintenance. using Fusion 360 allows me to design both CAD and PCB in one place, ensuring not only a more efficient design process, but also a better integration between the mechanical and electronic components of the robot.

Software

The overall software architecture of the robot is designed to be modular and scalable, allowing for easy integration of new features and functionalities in the future. The high logic that oversees the rest of the software are threads that run in parallel, each one is separated by functionality, for exampple one thread is responsible for face tracking, another is responsible for conversation management (TTS, STT and LLM logic), another one is responsible for self monitoring (sensors, and api statuses), and the last one to manage the las physical interface, a button that triggers specific actionswhen pressed. This way, if one thread crashes or has an error, the rest of the software can continue running without interruption, ensuring a more stable and reliable performance of the robot.

Of course each thread controll the corresponding hardware components, but before all start working, the system at each boot runs a self monitoring routine, where it checks the status of all the components and APIs, showing the process on the LCD and the back of the robot, this way we can ensure that everything is working correctly before starting the interactions, and if something is not working, the system can alert the user and prevent any potential issues during the interactions.

Due that the raspberry pi is not that powerful, and the software is quite heavy, I had to optimize the code as much as possible, for that I implemented APIs that connect to more powerful servers, in this case, my main computer, where I run the LLM server with LM Studio; a docker dev-container that handles SST with a light version of whisper, and TTS management, which connects to the official Kokoro Docker container.

This way, the raspberry pi can handle the basic interactions and hardware control, while the more heavy processing tasks are offloaded to more powerful servers, ensuring a smoother and more responsive performance of the robot.
